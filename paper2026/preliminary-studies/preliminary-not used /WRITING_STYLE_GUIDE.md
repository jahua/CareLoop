# Writing Style Guide for Academic Documentation

## Avoiding Repetitive Phrasing

### Problem: Checklist-Style Lists
**Avoid:**
```
- Removed redundant explanations
- Removed duplicate overview
- Removed large table
- Removed code examples
```

**Better:**
```
- Eliminated redundant explanations, consolidating theory into unified narrative
- Migrated duplicate overview to Section 2.2
- Transferred large table to Appendix C for cleaner flow
- Relocated code examples to Appendix A, maintaining readability
```

---

## Verb Variation for Similar Actions

### Moving/Removing Content

Instead of repeating "removed," use:
- **Eliminated** (for redundancy)
- **Consolidated** (for merging)
- **Migrated** (for moving to appendix)
- **Relocated** (for repositioning)
- **Transferred** (for shifting sections)
- **Streamlined** (for simplifying)
- **Merged** (for combining)

**Examples:**
- "Eliminated redundant explanations"
- "Consolidated technical specifications into Section 2.2"
- "Migrated node specification table to Appendix C"
- "Relocated EMA algorithm to Appendix A"
- "Transferred PostgreSQL schema details to Appendix C"
- "Streamlined metrics table into concise bullets"

---

### Adding/Creating Content

Instead of repeating "added," use:
- **Introduced** (for new concepts)
- **Established** (for frameworks)
- **Created** (for new sections)
- **Incorporated** (for integrating)
- **Integrated** (for weaving together)
- **Developed** (for building out)

**Examples:**
- "Introduced webhook-based architecture"
- "Established clear cross-references to appendices"
- "Created unified psychological foundation section"
- "Incorporated Devdas validation into theoretical grounding"

---

### Improving/Enhancing Content

Instead of repeating "improved," use:
- **Enhanced** (for making better)
- **Refined** (for polishing)
- **Strengthened** (for making more robust)
- **Clarified** (for making clearer)
- **Optimized** (for efficiency)

**Examples:**
- "Enhanced readability through appendix separation"
- "Refined terminology for consistency"
- "Strengthened theoretical grounding"
- "Clarified cross-references throughout"

---

## Combining Related Points

### Before (Repetitive):
```
**Key Improvements:**
- Removed redundant Big Five explanations
- Removed redundant Zurich Model explanations
- Removed duplicate architecture overview
```

### After (Combined):
```
**Key Improvements:**
- Eliminated redundant theoretical explanations (Big Five, Zurich Model), consolidating foundations into unified narrative
- Merged duplicate architecture overview into Section 2.2 for single authoritative reference
```

---

## Adding Descriptive Context

### Before (Bare):
```
- Removed table → Appendix C
```

### After (Contextual):
```
- Migrated N8N node specification table (14 rows) to Appendix C for cleaner methodology flow
```

**Benefits:**
- Explains *why* (cleaner flow)
- Quantifies impact (14 rows)
- Shows destination (Appendix C)

## Action Phrases for Documentation Changes

| Action | Verb Choices | Example |
|--------|-------------|---------|
| **Delete** | Eliminated, Removed, Stripped, Purged | "Eliminated redundant explanations" |
| **Move** | Migrated, Relocated, Transferred, Shifted | "Migrated table to Appendix C" |
| **Combine** | Consolidated, Merged, Unified, Integrated | "Consolidated specs into Section 2.2" |
| **Simplify** | Streamlined, Condensed, Simplified, Reduced | "Streamlined table into bullet points" |
| **Add** | Introduced, Established, Created, Added | "Introduced webhook architecture" |
| **Improve** | Enhanced, Refined, Strengthened, Optimized | "Enhanced readability" |
| **Keep** | Retained, Preserved, Maintained | "Retained concise descriptions" |

## Academic Tone Markers

### Add Purpose/Benefit
```
"Migrated table to Appendix C"
→ "Migrated table to Appendix C for cleaner methodology flow"

"Removed code snippets"
→ "Relocated code snippets to Appendix A, maintaining main text readability"
```

### Add Quantification
```
"Removed large table"
→ "Transferred 14-row node specification table to Appendix C"

"Consolidated sections"
→ "Merged 3 subsections (~850 words) into 2 sections (~600 words)"
```

### Add Cross-References
```
"Moved details to appendix"
→ "Transferred implementation details to Appendix A with explicit cross-references"
```

## Strategic Use of Visual Emphasis

### Checkmarks (✅) - Use Sparingly

**Problem:** Excessive checkmarks reduce their impact and create visual clutter.

**Guideline:** Reserve checkmarks for **critical milestones only**:
- ✅ Major targets achieved (word count goals, completion status)
- ✅ Key quality gates passed (validation complete, tests passing)
- ✅ Important deliverables (final files generated)

**Use standard bullets for:**
- Regular improvements
- Process changes
- Technical details
- Routine updates

### Before (Overused Checkmarks):
```
**Key Improvements:**
- ✅ Removed redundant explanations
- ✅ Consolidated technical specs
- ✅ Moved code to appendices
- ✅ Added cross-references
```

### After (Strategic Use):
```
**Key Improvements:**
- Eliminated redundant explanations
- Consolidated technical specs
- Relocated code to appendices
- Established cross-references

**Targets Met:**
- ✅ Word count: 9,311 (within 8,000-10,000 range)
- ✅ DOCX generated successfully
```

## Quick Reference Checklist

Before submitting any summary/change document:

- [ ] No verb used more than twice in same list
- [ ] Each point includes *why* (purpose/benefit)
- [ ] Quantify where possible (word counts, table rows, etc.)
- [ ] Combine related points when logical
- [ ] Add descriptive context, not bare actions
- [ ] Use academic tone, not checklist style
- [ ] **Checkmarks only for critical milestones** (max 2-3 per document)

---

**Remember:** Documentation should flow like prose, not read like a task list. Every change should explain its purpose and benefit, not just its mechanical action. Visual emphasis (checkmarks, bold text) should highlight truly important points, not decorate every bullet.
